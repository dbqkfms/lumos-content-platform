# Pre-Mortem - LUMOS Site Security & Content Protection

기준 스킬:
- `pre-mortem`

## Tigers (실제 리스크)

### 1. 프리뷰 영상 무단 저장
- 설명: 사이트에 올라간 프리뷰는 브라우저 개발자도구, 네트워크 캐시, 화면 녹화로 저장될 수 있다.
- 분류: Launch-blocking 아님, Fast-follow
- 대응:
  - 공개본은 항상 프리뷰 전용 파일 사용
  - 해상도 제한
  - 워터마크 또는 식별 가능한 프리뷰 버전 사용
  - 원본과 공개본 저장 경로 분리

### 2. Vimeo 임베드만으로 보호된다고 오해할 위험
- 설명: Vimeo는 직접 다운로드를 어렵게 만들 수는 있지만, 완전한 DRM이 아니다.
- 분류: Tiger / Fast-follow
- 대응:
  - “완전 보호” 표현 금지
  - 사이트 문구는 `프리뷰 전용`, `원본은 계약 후 제공` 중심
  - 필요한 경우 domain privacy, embed 제한, 비공개 폴더, 링크 제한 설정

### 3. 송출 시스템이 완성된 것처럼 보이는 과장
- 설명: 현재는 납품형 + 운영지원형인데, CMS/송출 플랫폼처럼 보이면 신뢰 리스크가 생긴다.
- 분류: Launch-blocking
- 대응:
  - 사이트 문구에서 현재 가능한 범위를 분명히 표시
  - CMS/원격 송출은 “향후 확장”으로 표현

### 4. 문의 후 기대치 mismatch
- 설명: 고객은 바로 플레이어/운영 솔루션까지 되는 줄 알 수 있다.
- 분류: Tiger / Launch-blocking
- 대응:
  - 문의 폼과 FAQ에 현재 제공 범위 명시
  - 납품형 / 운영지원형 / 향후 송출형 구분

## Paper Tigers (과장된 우려)

### 1. 사이트에 작품이 보이면 바로 사업이 불가능해진다
- 판단: 과장
- 이유: 프리뷰 기반 B2B 라이브러리는 공개 쇼케이스 자체가 영업 자산이다. 문제는 공개 여부가 아니라 공개 방식이다.

### 2. Vimeo를 쓰면 무조건 위험하다
- 판단: 과장
- 이유: Vimeo는 상업용 프리뷰 용도로 충분히 쓸 수 있다. 다만 보호 수단이 “완전 보안”은 아니라는 점을 이해해야 한다.

## Elephants (팀이 더 자주 말해야 할 것)

### 1. 원본/프리뷰 자산 관리 원칙
- 질문: 공개용 파일, 납품용 파일, 수정본, 공간별 파생본을 어떻게 구분할 것인가?

### 2. CS 접점
- 질문: 수정 요청과 운영 문의는 THISGLOBAL, LUMOS 중 어디로 먼저 들어와야 하는가?

### 3. 납품 이후 계약 범위
- 질문: 1회 납품인지, 시즌 업데이트 포함인지, 수정 회수는 어떻게 되는가?

## Launch-Blocking Action Plan

| Risk | Mitigation | Owner | Priority |
|---|---|---|---|
| 송출 시스템 과장 표현 | 사이트 문구를 납품형/운영지원형 중심으로 정리 | Site owner | High |
| 문의 후 기대치 mismatch | 문의 폼/FAQ/Process에 현재 제공 범위 명시 | Site owner | High |
| 프리뷰 보안 오해 | 원본 보호 정책 문구 추가 | Content owner | High |

## 권장 정책 문구

- 공개 사이트의 영상은 프리뷰용입니다.
- 원본 및 실제 운영용 파일은 프로젝트 협의 후 별도 제공됩니다.
- 현재는 납품형과 운영지원형 중심이며, 송출/CMS 연동은 확장 단계입니다.
